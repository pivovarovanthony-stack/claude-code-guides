Claude Code Setup Guide

Step-by-step instructions to set up Claude Code on any project — from a simple script to a complex multi-language application. Works on Mac, Windows, and Linux.

Install Claude Code

Claude Code is Anthropic's official CLI tool. It runs in your terminal and works with any project.

Requirements

  • macOS 13+, Windows 10+, or Ubuntu 20.04+
  • 4GB+ RAM
  • Claude Pro, Max, Team, or Enterprise account
  • Internet connection

Install Command

macOS / Linux
curl -fsSL https://claude.ai/install.sh | bash
Windows (PowerShell)
irm https://claude.ai/install.ps1 | iex
! Do NOT install via npm — it's deprecated. The native installer auto-updates.

Verify Installation

claude --version

VS CodeExtension: search "Claude Code" in marketplace

Initialize Your Project

Navigate to your project folder and start Claude Code:

# Go to your project
cd ~/my-project

# Start Claude Code
claude

# Or auto-generate a starter CLAUDE.md
claude /init
* Tip: Running /init inside Claude Code auto-generates a CLAUDE.md by scanning your project. Good starting point, but you'll want to customize it.

If your project isn't a git repo yet:

git init
git add .
git commit -m "Initial commit"
3

Write Your CLAUDE.md

This is the most important file. It tells Claude how to work on your project. Think of it as the project's "brain" — Claude reads it at the start of every session.

Golden Rules
  • Keep it under 200 lines — longer files reduce instruction adherence
  • Be specific: "Use 4-space indentation" not "Format code properly"
  • Include verification details: "Run npm test before committing"
  • Use @filepath to import longer docs instead of inlining them
  • Remove conflicting instructions — audit periodically

File Types

FilePurposeGit?
CLAUDE.md Project instructions — shared with team Commit
CLAUDE.local.md Your personal preferences Gitignored
~/.claude/CLAUDE.md Your global preferences (all projects) Personal

Template — Copy & Customize

# Project Name

> One-line description of what this project does.

## Stack

| Layer | Tech |
|-------|------|
| Frontend | React / Next.js / Vue / etc. |
| Backend | Node / Python / PHP / etc. |
| Database | PostgreSQL / MySQL / MongoDB |
| Hosting | Vercel / Docker / AWS |

## Project Structure

@README.md
# Or describe it inline (keep it short)

## Coding Conventions

- 4 spaces indentation
- Single quotes for strings
- Semicolons required
- camelCase for variables, PascalCase for components
- Conventional commits: feat:, fix:, docs:, refactor:

## Commands

```
npm run dev       # Start dev server
npm test          # Run tests
npm run build     # Production build
npm run lint      # Lint code
```

## Rules

- Never hardcode credentials — use .env
- Don't push to remote without asking
- Run tests before committing
- @.claude/rules/security.md

Import Syntax

Reference other files instead of copy-pasting content:

# These pull in file content when Claude loads the project
@README.md
@docs/api-reference.md
@.claude/rules/security.md
@package.json  # Claude sees available npm commands

Project Structure

Here's the recommended folder layout for a Claude Code-powered project:

my-project/ ├── CLAUDE.md # Project instructions (commit to git) ├── CLAUDE.local.md # Personal preferences (gitignored) ├── .gitignore ├── .mcp.json # MCP server configs (gitignored) │ ├── .claude/ # Claude Code configuration │ ├── settings.json # Team-shared settings (commit) │ ├── settings.local.json # Your local overrides (gitignored) │ │ │ ├── rules/ # Modular coding standards │ │ ├── common/ # Language-agnostic rules │ │ │ ├── coding-style.md │ │ │ ├── testing.md │ │ │ ├── security.md │ │ │ └── git-workflow.md │ │ └── [language]/ # Language-specific overrides │ │ ├── coding-style.md │ │ ├── patterns.md │ │ └── testing.md │ │ │ ├── skills/ # Custom workflows & commands │ │ └── deploy/ │ │ ├── SKILL.md # Skill definition │ │ └── scripts/ │ │ │ └── agents/ # Custom subagents │ ├── planner.md │ └── code-reviewer.md │ └── src/ # Your actual code
i Auto-memory is stored separately at ~/.claude/projects/<project>/memory/ — you don't need to create it. Claude manages it automatically.

Rules — Coding Standards

Rules are markdown files in .claude/rules/ that define coding standards. Claude loads them automatically based on file paths.

How Rules Work

  • Rules in common/ apply to all files
  • Language-specific rules override common ones when there's a conflict
  • You can use paths: frontmatter to limit when a rule activates

Path-Specific Rules

---
paths:
  - "src/api/**/*.ts"
  - "src/**/*.{ts,tsx}"
---

# API Development Rules
- All endpoints must include input validation
- Use Zod schemas for request/response types
- Include OpenAPI documentation comments

Pre-Built Rule Sets (everything-claude-code)

You can install battle-tested rules from the everything-claude-code repo:

# Clone to temp, copy what you need, delete
git clone --depth 1 https://github.com/affaan-m/everything-claude-code.git /tmp/ecc
cp -r /tmp/ecc/rules/common .claude/rules/
cp -r /tmp/ecc/rules/python .claude/rules/   # Your language
cp -r /tmp/ecc/rules/typescript .claude/rules/
rm -rf /tmp/ecc

Available languages: TypeScript, Python, PHP, Java, Go, Rust, Kotlin, C++, C#, Swift, Dart, Perl

Skills — Custom Commands

Skills are reusable workflows you invoke with /skill-name. They live in .claude/skills/.

Create a Skill

# File: .claude/skills/deploy/SKILL.md

---
name: deploy
description: Deploy the app to production
allowed-tools: "Bash(npm *) Bash(docker *) Read"
---

Deploy to production:

1. Run the test suite with `npm test`
2. Build with `npm run build`
3. Deploy with `docker-compose up -d`
4. Verify deployment is healthy

Skill Frontmatter Options

OptionWhat It Does
disable-model-invocation: trueOnly YOU can trigger it (not Claude on its own)
user-invocable: falseOnly Claude can use it (background knowledge)
allowed-toolsTools Claude can use without asking permission
context: forkRun in isolated subagent (protects main context)
agent: ExploreWhich agent type to use (Explore, Plan, general-purpose)

Skill with Support Files

.claude/skills/seo-audit/ ├── SKILL.md # Overview (keep under 500 lines) ├── reference.md # Detailed docs (loaded on demand) ├── examples.md # Usage examples └── scripts/ └── check-rankings.py

Agents — Specialized Workers

Agents are markdown files in .claude/agents/ that define specialized subagents Claude can delegate work to.

# File: .claude/agents/code-reviewer.md

---
name: code-reviewer
description: Reviews code changes for quality, security, and best practices
tools: Read, Grep, Glob, Bash
---

You are a code reviewer. When asked to review changes:

1. Read all modified files
2. Check for security issues (hardcoded secrets, SQL injection, XSS)
3. Check for code quality (naming, complexity, duplication)
4. Check for test coverage
5. Provide actionable feedback with file:line references

Hooks — Automation

Hooks are shell commands that run automatically in response to Claude Code events. Configure them in .claude/settings.json.

Hook Events

EventWhen It FiresCan Block?
PreToolUseBefore a tool runsYes (exit 2)
PostToolUseAfter a tool runsNo
StopWhen Claude finishes respondingNo
SessionStartWhen session beginsNo
SessionEndWhen session endsNo
SubagentStartWhen a subagent launchesNo
FileChangedWhen a file is modifiedNo

Example: Block Dangerous Git Commands

// In .claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "if": "Bash(git push --force*)",
            "command": "echo 'BLOCKED: force push not allowed' && exit 2",
            "timeout": 5000
          }
        ]
      }
    ]
  }
}
i Start simple. You don't need hooks on day one. Add them as you discover repeated pain points.

Settings — Permissions & Config

Settings Hierarchy (all loaded together)

ScopeFileWho
1. Managed/Library/Application Support/ClaudeCode/IT/Admin (unoverrideable)
2. User~/.claude/settings.jsonYou (all projects)
3. Project.claude/settings.jsonTeam (commit to git)
4. Local.claude/settings.local.jsonYou (gitignored)

Recommended Starter Settings

// .claude/settings.json (commit to git)
{
  "autoMemoryEnabled": true,
  "permissions": {
    "deny": [
      "Bash(rm -rf *)",
      "Edit(.env)",
      "Edit(.env.*)"
    ]
  }
}
// .claude/settings.local.json (gitignored — your personal overrides)
{
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(npm *)",
      "Read(**)"
    ]
  }
}

MCP — External Tool Integrations

MCP (Model Context Protocol) lets Claude connect to external services — GitHub, databases, Figma, Google Sheets, Notion, and more.

Configure in .mcp.json

{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_PERSONAL_ACCESS_TOKEN": "$GITHUB_TOKEN"
      }
    }
  }
}

Add via CLI (Recommended)

# Add a user-level MCP server
claude mcp add -s user my-server-name -- npx -y @some/mcp-server

# Add a project-level MCP server
claude mcp add -s project my-server -- npx -y @some/mcp-server
! Keep under 10 MCP servers enabled. Each adds ~5-10K tokens of overhead to every conversation.

Memory — Persistent Knowledge

Claude automatically remembers things across sessions. Memory is stored at ~/.claude/projects/<project>/memory/.

What Claude Remembers Automatically

  • Build commands it discovers
  • Debugging insights
  • Architecture notes
  • Your code style preferences
  • Workflow patterns

Ask Claude to Remember

You can explicitly tell Claude to save things:

"Remember that our staging server is at staging.example.com"
"Remember to always run tests before pushing"
"Remember that the billing module uses Stripe webhooks, not polling"

Disable Auto-Memory

// In settings.json
{ "autoMemoryEnabled": false }

Setup Checklist

Use this checklist to verify your project is properly set up:

Essential (Do First)
  • Install Claude Code (native installer, not npm)
  • Initialize git repo in project
  • Create CLAUDE.md with project overview, stack, conventions
  • Create .gitignore (include CLAUDE.local.md, settings.local.json, .env, .mcp.json)
  • Run claude in project root — verify it reads your CLAUDE.md
Recommended (Do Next)
  • Create CLAUDE.local.md with personal preferences
  • Create .claude/settings.json with deny rules for destructive commands
  • Add coding rules to .claude/rules/ (from everything-claude-code or custom)
  • Set up .claude/settings.local.json with your permission overrides
  • Verify auto-memory is enabled
Advanced (When Ready)
  • Add specialized agents to .claude/agents/
  • Create project-specific skills in .claude/skills/
  • Configure hooks in settings.json (start with git safety)
  • Set up MCP servers for external tools (GitHub, Notion, etc.)
  • Create ~/.claude/CLAUDE.md for global personal preferences
  • Set up cross-project references in CLAUDE.local.md

What NOT to Put in CLAUDE.md
  • Credentials, API keys, tokens (use .env)
  • Entire file contents (use @imports instead)
